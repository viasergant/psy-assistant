# Introduction

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

This plan defines the implementation sequence for PA-23 client master profile management in the psy-assistant monorepo. The plan is constrained to the existing Spring Boot backend and Angular clients feature. The first executable slice extends the current read-only client detail flow into a writable client profile for personal information, contact details, one emergency contact, and communication preferences. Tags and profile photo upload are implemented in a second slice after the backend audit, authorization, and optimistic locking model is stable.

## 1. Requirements & Constraints

- **REQ-001**: Implement client profile sections for Personal Information, Contact Details, Referral Information, Emergency Contact, Communication Preferences, Tags and Case Type, and Profile Photo.
- **REQ-002**: Expose `GET /api/v1/clients/{id}` for the full client profile read model.
- **REQ-003**: Expose `PUT /api/v1/clients/{id}` for structured profile updates.
- **REQ-004**: Expose `PATCH /api/v1/clients/{id}/tags` for tag replacement.
- **REQ-005**: Expose `POST /api/v1/clients/{id}/photo` for profile photo upload.
- **REQ-006**: Record immutable field-level audit entries for every successful profile mutation.
- **REQ-007**: Assigned therapists can read the client profile but cannot edit PA-23 fields.
- **REQ-008**: Reception/admin staff and system administrators can read and edit PA-23 profile fields.
- **REQ-009**: Supervisors can read all PA-23 profile sections and cannot mutate them.
- **REQ-010**: Tags are free text with hints from existing values.
- **REQ-011**: Internal client identifier must be auto-generated by the system.
- **SEC-001**: Do not log personal or sensitive client data in application logs.
- **SEC-002**: Enforce authorization on the server; frontend capability flags are advisory only.
- **SEC-003**: Accept profile photo uploads only for `image/jpeg` and `image/png` with a maximum size of `5242880` bytes.
- **CON-001**: Keep the implementation inside `backend/src/main/java/com/psyassistant/crm/clients` and `frontend/src/app/features/clients`.
- **CON-002**: Do not introduce a new microservice or a new top-level domain module.
- **CON-003**: Activity timeline and search remain out of scope for PA-23 and belong to PA-24.
- **CON-004**: Standalone client creation outside the lead conversion flow remains out of scope.
- **CON-005**: Do not add optimistic locking to `BaseEntity`; add versioning to `Client` only.
- **GUD-001**: Use Flyway migrations as the only source of schema evolution.
- **GUD-002**: Keep API contracts explicit with request and response DTOs.
- **GUD-003**: Preserve existing lead-to-client conversion behavior implemented in `V7__lead_to_client_conversion.sql`.
- **PAT-001**: Use dedicated audit tables for client profile changes instead of extending the current generic audit log.
- **PAT-002**: Use a storage abstraction for profile photos with a local filesystem implementation first.
- **PAT-003**: Return server-computed capability flags in the client profile response.
- **DEC-001**: Client identifier format for PA-23 is `CL-<8 uppercase hex chars from UUID>`, derived once at creation time and persisted in the database.
- **DEC-002**: Therapist editability for PA-23 slice one is `read-only for all profile sections`.
- **DEC-003**: The first photo storage implementation uses a configurable local filesystem root declared in application configuration.

## 3. Alternatives

- **ALT-001**: Store client profile fields as a single JSON column in `clients`. Rejected because field-level auditing, validation, and later search integration would be weaker and more expensive to evolve.
- **ALT-002**: Store profile photos directly in PostgreSQL as binary content. Rejected because operational cost, payload size, and content serving complexity are unnecessary for the current architecture.
- **ALT-003**: Reuse the existing generic audit log for field history. Rejected because it does not provide the structured per-field old/new storage required by PA-23.
- **ALT-004**: Implement tags and photo upload in the same slice as the first writable profile API. Rejected because it would couple authorization, audit, storage, and normalization concerns into one migration and one unstable DTO contract.

## 7. Risks & Assumptions

- **RISK-001**: Adding read permissions incorrectly could broaden therapist or supervisor access outside the intended assignment rules.
- **RISK-002**: Audit tables may grow quickly if profile changes are frequent; indexes and append-only access patterns must remain efficient.
- **RISK-003**: The new `client_code` format must remain immutable after generation or downstream references may drift.
- **RISK-004**: Introducing `@Version` on `Client` may require careful handling in tests and DTO mapping to avoid false-positive conflicts.
- **RISK-005**: Existing pre-conversion notes in `Client.notes` may be mistaken for therapist notes unless the UI labels them clearly.
- **ASSUMPTION-001**: `Client` remains the authoritative aggregate for PA-23 and no separate therapist-note entity is required in this ticket.
- **ASSUMPTION-002**: Assigned therapist identity can be represented by `assigned_therapist_id` on `clients` without requiring a join table in PA-23.
- **ASSUMPTION-003**: Local filesystem storage is acceptable for the initial profile photo implementation in non-production environments.
- **ASSUMPTION-004**: The existing authentication layer exposes a stable user identifier suitable for audit actor attribution and therapist assignment checks.

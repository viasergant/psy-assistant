## Context

The psy-assistant CRM handles sensitive psychological client data. Before any domain feature is released to a real environment, the application must emit machine-parseable logs, expose health and metrics signals, and be ready to connect to an APM platform without requiring source-code changes. Retrofitting observability onto an already-built system is significantly more expensive than establishing it at the foundation layer.

---

## Decision

### 1. JSON log format (logstash-logback-encoder)

**Chosen library:** `net.logstash.logback:logstash-logback-encoder:8.0`

**Rationale:**
- Produces a single-line JSON object per log event, compatible with ELK Stack (Elasticsearch/Logstash/Kibana), Datadog Logs, and Google Cloud Logging without a custom parser.
- The logstash schema (`@timestamp`, `level`, `logger_name`, `message`, `thread_name`) is the de-facto standard for Java services and is understood natively by every major log aggregation platform.
- Profile-aware configuration in `logback-spring.xml` (using `<springProfile>`) ensures that local and test environments continue to emit human-readable output while production emits structured JSON.

**Profile mapping:**

| Profile | Format | Appender |
|---------|--------|----------|
| `local`, `test` | Human-readable pattern | `ConsoleAppender` |
| `prod` | JSON (logstash schema) | `AsyncAppender` â†’ `ConsoleAppender` + `RollingFileAppender` |

**Important:** The file must be named `logback-spring.xml` (not `logback.xml`) so that Spring Boot injects profile resolution before Logback initialises. Using `logback.xml` silently breaks `<springProfile>` tags.

### 2. Changing log levels

Log levels are configured per environment in the Spring profile YAML files. **No source-code change or redeploy is required** to change a log level in a running application if Spring Boot Admin or the Actuator `/actuator/loggers` endpoint is available. For a config-file change:

| Environment | File | Root level | `com.psyassistant` level |
|-------------|------|-----------|--------------------------|
| `local` | `application-local.yml` | `DEBUG` | `DEBUG` |
| `prod` | `application-prod.yml` | `WARN` | `INFO` |
| default | `application.yml` | `INFO` | `INFO` |

To change a level, edit the relevant `logging.level.*` key in the environment YAML and restart the service (or use the Actuator `/actuator/loggers` endpoint at runtime if exposed).

### 3. Attaching an APM agent

No application source-code change is needed. The Java agent is attached at JVM startup via the `-javaagent` flag:

```bash
java \
  -javaagent:/opt/apm/dd-java-agent.jar \
  -Ddd.service=psy-assistant \
  -Ddd.env=prod \
  -Ddd.version=1.0.0 \
  -jar psy-assistant-backend.jar
```

Replace `dd-java-agent.jar` with the agent JAR for your chosen APM platform (Datadog, New Relic, Dynatrace, OpenTelemetry Java agent, etc.). The application does not need to know which agent is attached.

#### Adding a pull-based Micrometer registry

If using a pull-based platform (e.g., Prometheus, Grafana), add the corresponding Micrometer registry dependency to `pom.xml`:

```xml
<!-- Already included for Prometheus -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```

For a push-based registry (e.g., Datadog):

```xml
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-datadog</artifactId>
</dependency>
```

Then configure the registry in `ObservabilityConfig` using a `MeterRegistryCustomizer` bean (example code is already present as a comment in that class). No other source changes are needed.

### 4. PII masking policy for log statements

**Policy:** Log lines must never include raw personal data. This is a compliance requirement applicable to all profiles (local, staging, production).

**What is allowed:**
- Opaque identifiers: `clientId=42`, `sessionId=abc-123-def`, `userId=7`
- Status codes, durations, metric values
- Stack traces (these do not contain client data by design)
- Request IDs (UUIDs generated by `CorrelationIdFilter`)

**What is strictly prohibited in log output:**
- Client names or any part of a name
- Contact details (email address, phone number, address)
- Session notes or any free-text content entered by a therapist or client
- Diagnosis codes or clinical identifiers linked to a specific person
- Any data that could identify a natural person (GDPR Article 4(1))

**Enforcement:**
- Developers must review all new log statements before merge to confirm compliance.
- Security review sign-off is required as part of the Definition of Done for every story that adds new log output.
- Log aggregation platform ingestion rules should be configured with PII-detection alerts as a defence-in-depth measure (out of scope for this ADR).

**Regulatory context:** The organisation is subject to GDPR-equivalent requirements. Sensitive health data (Article 9 special category data) is processed. The masking policy applies to all environments; there is no "debug exception" for production-like data in any environment.

## Consequences

- Structured log output is immediately compatible with ELK, Datadog, and GCP Logging.
- The `requestId` MDC field links all log lines for a single HTTP request, enabling fast incident diagnosis.
- `/actuator/health` is publicly accessible (required for load balancer health checks); all other sensitive Actuator endpoints (`/actuator/env`, `/actuator/beans`) are blocked unconditionally.
- Prometheus metrics are available at `/actuator/prometheus` for scraping; HikariCP connection pool metrics are auto-registered when the Prometheus dependency is on the classpath.
- APM can be attached at deployment time without touching application code.
- PII masking must be enforced by developer discipline and code review; no automated redaction is in place at this time.
